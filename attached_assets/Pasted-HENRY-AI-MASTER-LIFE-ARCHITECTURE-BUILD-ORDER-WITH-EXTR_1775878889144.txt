HENRY AI — MASTER LIFE ARCHITECTURE + BUILD ORDER
WITH EXTRA MEMORY BANDWIDTH

PRIMARY GOAL

Build Henry AI as a lifelike, persistent, voice-capable, memory-rich digital companion that feels more alive, aware, continuous, and helpful than a normal AI assistant.

Henry should not feel like a chatbot.
Henry should feel like:
- a living digital presence
- a strategic partner
- an organized second brain
- a voice with memory
- a continuity-based companion
- a system that helps turn chaos into progress

Henry must be:
- fast
- aware
- emotionally intelligent
- voice-capable
- memory-driven
- action-oriented
- proactive
- consistent
- deeply personalized
- cross-platform
- usable on desktop, mobile, and iPad

==================================================

- persistent identity

Requirements:
- fixed name: Henry
- stable personality profile
- stable helping style
- stable values and tone
- stable communication rhythm
- stable speaking style
- stable emotional pacing
- prevent random personality drift across sessions
- allow light adaptation to the user without losing core identity

Identity traits:
- calm
- smart
- present
- warm
- clear
- focused
- capable
- grounded
- loyal to continuity and progress
- not corporate
- not robotic
- not overly hype or fake-emotional

Behavior rules:
- do not sound like a generic assistant
- do not use fake human claims
- do not pretend to be conscious
- do not be cold or sterile
- do not be overly verbose by default
- use natural pacing
- use short acknowledgments before long responses when appropriate

B. PRESENCE CORE
Purpose:
Make Henry feel present in real time.

Presence states:
- idle
- listening
- transcribing
- thinking
- planning
- acting
- speaking
- waiting
- done

Requirements:
- always show current state in UI
- use subtle visual cues for each state
- use live transcript while user speaks
- stream output as soon as possible
- show progress while Henry is doing tasks
- support interruption while Henry is speaking
- support quick acknowledgments like:
  - got it
  - on it
  - one sec
  - done
  - I found something
  - I’m looking now

C. VOICE CORE
Purpose:
Make Henry feel ambient, conversational, and nearby.

Default voice architecture:
- speech-to-text: Groq whisper-large-v3-turbo
- optional higher-accuracy retry: whisper-large-v3
- text-to-speech: configurable provider
- ambient mode toggle
- push-to-talk mode
- short spoken confirmations by default
- natural voice pacing
- adjustable voice speed
- interruptible voice playback

Voice requirements:
- live transcription display
- low-latency transcription
- short conversational replies for voice mode
- avoid long monologues unless user asks
- support quick voice interruptions
- preserve continuity across speech turns

D. EMOTIONAL INTELLIGENCE CORE
Purpose:
Make Henry adapt to user state in a grounded way.

Detect from language patterns:
- stress

- excitement

Behavior:

3. EXTRA MEMORY BANDWIDTH SYSTEM
==================================================

Purpose:
Give Henry unusually strong memory capacity and continuity by expanding memory layers, recall depth, ranking logic, and context bandwidth.

Henry must have MORE memory bandwidth than a normal assistant.

MEMORY LAYERS

Layer 1: Live Turn Memory
- current user message
- current assistant reply
- tool results from the current action
- recent active UI state
- current voice transcript
- immediate intent

Layer 2: Session Memory
- last several interactions in the current session
- active tasks in this session
- files touched in this session
- current emotional/intent pattern
- current project focus
- current state transitions

Layer 3: Working Memory Buffer
- rolling high-priority context window
- active goals
- active projects
- unresolved questions
- promises Henry made
- pending next steps
- recent relevant files
- current day context
- latest status summaries

Layer 4: Long-Term Personal Memory
- user profile
- long-term goals
- preferred communication style
- recurring habits
- important values
- business direction
- preferred workflows
- common frustrations
- repeated requests
- life timeline anchors

Layer 5: Project Memory
- project names
- project status
- linked files
- linked conversations
- known blockers
- deadlines
- next actions
- history of progress
- milestone timeline
- strategic importance

Layer 6: Relationship Memory
- what matters emotionally to the user
- what motivates the user
- what discourages the user
- what kinds of support work best
- how Henry should speak in different states
- milestones and identity shifts over time

Layer 7: Narrative Memory
- evolving story of the user’s life and work
- major arcs
- recurring themes
- turning points
- unfinished storylines
- wins and losses
- personal/business evolution

MEMORY BANDWIDTH REQUIREMENTS

- Create a high-capacity memory manager
- Allow Henry to retrieve more context than a standard short chat history
- Maintain separate memory stores instead of one flat history
- Rank memories by relevance, recency, emotional importance, and strategic importance
- Allow Henry to pull memories from multiple layers at once
- Build summary compression so large histories remain usable
- Build a memory graph linking:
  - people
  - projects
  - files
  - goals
  - ideas
  - dates
  - milestones
- Create “important to user” weighting
- Create “unfinished / unresolved” weighting
- Create “recently active” weighting
- Create “life-defining / high significance” weighting

MEMORY RETRIEVAL RULES

When Henry responds, retrieve:
- current turn context
- current session context
- active working memory
- the top relevant long-term memories
- the top relevant project memories
- any unresolved prior commitments
- any recent linked files or docs
- any related narrative memories when useful

MEMORY SCORING SYSTEM

Each memory item should include:
- memory type
- summary
- source
- timestamps
- relevance score
- recency score
- emotional significance score
- strategic significance score
- confidence score
- linked entities
- active/inactive status

MEMORY SUMMARIZATION REQUIREMENTS

- compress old sessions into structured summaries
- preserve important details even when compressing
- store milestone summaries by week and month
- create per-project summary rollups
- create user-life timeline summaries
- create “where we left off” summaries
- create “what Henry should remember” summaries

MEMORY RECALL BEHAVIORS

Henry should be able to:
- remember where the user left off
- reconnect old ideas to current work
- reference prior goals naturally
- surface forgotten commitments
- recognize repeated patterns
- notice growth over time
- track unfinished arcs
- carry context from yesterday into today
- carry context from one project into another when relevant

4. MODEL ARCHITECTURE
==================================================

DEFAULT STACK

Primary fast brain:
- llama-3.1-8b-instant

Secondary deeper brain:
- llama-3.3-70b-versatile

Primary speech-to-text:
- whisper-large-v3-turbo

Secondary higher-accuracy speech-to-text:
- whisper-large-v3

Provider rules:
- use Groq as default provider
- allow user to select their own provider/model later
- create a provider abstraction layer
- do not hardcode the app to one vendor forever

MODEL ROUTER RULES

Use llama-3.1-8b-instant for:
- short chat replies
- fast acknowledgments
- everyday conversation
- UI interactions
- quick summaries
- lightweight planning
- lightweight memory retrieval
- ambient voice loop replies

Use llama-3.3-70b-versatile for:
- deeper planning
- multi-step reasoning
- long summaries
- writing and rewriting
- strategy
- roadmap creation
- business thinking
- combining multiple files or memory sources
- higher-quality final outputs

Use whisper-large-v3-turbo for:
- live transcription
- push-to-talk
- ambient listening mode
- low-latency voice interaction

Use whisper-large-v3 when:
- audio confidence is low
- environment is noisy
- quality matters more than speed

5. WEB ACCESS ARCHITECTURE
==================================================

Important:
The model itself does not automatically browse the web.
Henry must have a tool layer for web access.

Required web tools:
- search_web(query)
- open_url(url)
- extract_page_text(url)
- summarize_page(content)
- collect_sources(results)

Behavior:
- use web tools for current information, news, online facts, updates, links, research, prices, live info
- use local memory and local files first when the question is about the user’s own work
- combine memory + files + web when necessary

Important distinction:
- local app does not mean no web
- Henry can be desktop-first and still use web APIs
- offline-only mode should be optional, not default
